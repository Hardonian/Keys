## One-Sentence Definition

**KEYS is the keyring to the modern toolshed—a marketplace of structured, validated, reusable assets (notebooks, prompts, workflows, playbooks, guides) that unlock practical, repeatable, commercial capability in external tools without competing with them.**

---

## The Toolshed Metaphor (Explained Once, Never Again)

Imagine a modern toolshed filled with powerful tools: Cursor, Jupyter, GitHub, Stripe, Supabase, AI Studio, and countless others. Each tool provides raw capability—but capability alone isn't enough. You need to know how to use them effectively.

**KEYS is the keyring.**

Just as a physical keyring holds keys that unlock doors, KEYS holds structured assets that unlock capability in digital tools. The tool provides the power. The key provides the leverage.

**This metaphor is complete. It will not be re-explained.**

---

## What KEYS Is

### A Marketplace of Keys
KEYS hosts a curated marketplace of keys organized by:
- **Tool** (Cursor Keys, Jupyter Keys, GitHub Keys, Stripe Keys, etc.)
- **Outcome** (Automation Keys, Monetization Keys, Validation Keys, etc.)
- **Maturity** (Starter Keys, Operator Keys, Scale Keys)

### Tool-Agnostic
KEYS doesn't care which AI model you use, which cloud provider you prefer, or which framework you choose. It provides keys that work with tools you already own.

### Outcome-Driven
Every key unlocks a specific, practical outcome. Not "AI assistance" but "unlock Stripe subscription management using this workflow."

### Reusable Assets
Keys are structured, validated, reusable, and updatable. They are not one-off solutions but permanent unlocks.

### Future-Proof
As tools evolve, KEYS evolves with them. New tools mean new keys. KEYS doesn't compete with tools—it amplifies them.

## What KEYS Is NOT

### ❌ An AI Model
KEYS is not an AI model, AI agent, or AI companion. It doesn't generate code, answer questions, or replace tools. It unlocks capability in tools that already exist.

### ❌ An AI Wrapper
KEYS is not a wrapper around AI models. It doesn't provide AI functionality—it provides keys that unlock AI tools.

### ❌ A Dev Tool
KEYS is not a development tool. It doesn't replace IDEs, editors, or development environments. It provides keys to unlock them.

### ❌ A Notebook Platform
KEYS is not a notebook platform. It doesn't host notebooks or provide notebook infrastructure. It provides notebook keys that unlock Jupyter.

### ❌ A Prompt Manager
KEYS is not a prompt manager. It doesn't store and organize prompts for convenience. It provides prompt keys that unlock tools.

### ❌ A Competitor to Cursor, Claude, GitHub, Jupyter, etc.
KEYS does not compete with any tool vendor. It amplifies them by providing keys that unlock their capabilities.

### ❌ A Platform Lock-In
KEYS doesn't require you to use specific platforms or services. Keys work with tools you already own.

### ❌ A Generic Solution
KEYS doesn't provide vague "AI assistance." Every key unlocks a specific, practical outcome in a specific tool.

### ❌ A One-Time Product
KEYS assets are reusable, updatable, and shareable. They're keys, not one-off solutions.

If AI development stops tomorrow, KEYS still makes perfect sense:

1. **Tools Still Exist**: Cursor, Jupyter, GitHub, Stripe, Supabase, and other tools continue to exist and evolve.

2. **Capability Still Needs Unlocking**: Users still need to know how to use tools effectively, even without AI acceleration.

3. **Patterns Still Matter**: Proven patterns, workflows, and best practices remain valuable regardless of AI progress.

4. **Outcomes Still Matter**: Practical outcomes (monetization, automation, validation) remain valuable regardless of AI progress.

5. **Keys Still Unlock**: Keys unlock capability in tools, not AI models. Tools persist even if AI stagnates.

**KEYS is tool-focused, not AI-focused.**

## Decision Framework

When evaluating any new feature, asset, or direction, ask:

1. **Does this unlock real capability in an external tool?**
   - If no, it's not a key.

2. **Does this compete with a tool?**
   - If yes, it's not a key.

3. **Does this produce a tangible, reusable output?**
   - If no, it's not a key.

4. **Is this tool-agnostic?**
   - If no, reconsider.

5. **Does this optimize for usefulness over novelty?**
   - If no, reconsider.

6. **Does this violate any non-goals?**
   - If yes, reject it.
